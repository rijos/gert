Configuration

Every server-level knob, where it binds from, and what it defaults to. Standard ASP.NET Core layering applies - last one wins:

appsettings.json -> appsettings.{Environment}.json -> environment variables -> command line
  • Environment variables - replace : with __: Gert__Embeddings__Parameters__BaseUrl=http://vllm:8000. Map entries by key: Gert__Chat__Providers__qwen36-thinking__Parameters__Model=qwen36.
  • Command line - dotnet run --project src/Gert.Api -- --Gert:Embeddings:Parameters:BaseUrl=... (highest precedence; beats launch-profile env vars too).
  • Secrets never go in appsettings.json - a bearer key, if your upstream needs one, arrives via env vars or dotnet user-secrets. Chat keys are per provider (Gert:Chat:Providers:<slug>:Parameters:ApiKey); the embeddings key is Gert:Embeddings:Parameters:ApiKey.

The whole Gert:* tree follows one rule: functionality -> choose an implementation (Type) -> configure it (Parameters). Cross-implementation knobs sit beside Type; the connection / impl-private config lives under Parameters. HTTP resilience (RequestTimeoutSeconds/RetryCount) is per item that makes calls - each chat provider's Parameters and Gert:Embeddings:Parameters carry their own; there is no shared HTTP section.

Gert:Embeddings - the embeddings upstream

The embeddings functionality. Type selects the implementation (OpenAI only today - an OpenAI-compatible /v1/embeddings upstream, vLLM serving bge-m3 in the reference deployment); an unknown Type fails fast at startup. The connection + resilience live under Parameters.

KeyDefaultNotes
Parameters:BaseUrlhttp://localhost:8000Embeddings server base URL, without /v1.
Parameters:ApiKey(unset)Secret - env / user-secrets only. Sent as Authorization: Bearer. (Each chat provider carries its own key.)
Parameters:Modelbge-m3Sent as model on /v1/embeddings. Knowledge upload (RAG) fails if the upstream doesn't serve it.
Parameters:Dimensions1024Must match the embedding model (bge-m3 = 1024). Effectively immutable once data exists - it bakes into every rag.db.
Parameters:RequestTimeoutSeconds120Per-attempt timeout (time to response headers) for the embeddings path's own resilience.
Parameters:RetryCount2Retries on transient pre-stream failure. Embedding POSTs are idempotent, so retries are safe. 0 disables.

vLLM prefix caching. Gert's chat requests are prefix-cache friendly: a static system prompt per project, history replayed verbatim, deterministic tool specs, no per-request unique fields. Make sure your vLLM has automatic prefix caching enabled (it's on by default in the V1 engine) and watch vllm:prefix_cache_hit_rate.

Gert:Chat:Providers - the chat provider catalog

A map keyed by provider slug - each entry is one named chat preset, and the slug is the id that GET /api/models publishes to the picker (in configured/document order). When the section is absent or empty, the catalog falls back to a single default openai provider built from Gert:Embeddings:Parameters:BaseUrl (single-vLLM zero-config boot). The picker's default is Gert:Chat:DefaultProvider - a chat-level key (a sibling of Providers) naming a slug; unset falls back to the first entry in document order, and a value matching no configured slug is a startup error.

The split is catalog/capabilities on the entry, connection + sampling + per-item resilience under Parameters:

KeyRequiredNotes
NamenoDisplay name in the picker. Defaults to the slug.
TypenoSelects the chat-client impl. "openai" (default) is the only one implemented - an OpenAI-compatible / vLLM endpoint; the schema is open for others.
CapabilitiesnoCapability tokens, shown as badges. "tools" is load-bearing - it gates tool calling. Unset means permissive; an explicit list without "tools" disables tools for that provider. "vision" and others are display-only today.
ContextyesContext window in tokens (the "128K ctx" badge) and the budget an inline message attachment is gated against. vLLM reports it as max_model_len. A configured provider with no positive Context fails closed at startup.
FastnoDisplay-only "- fast" marker.
ParametersyesThe Type-specific connection + sampling + resilience bag (see below). The picker's endpoint hint is taken from Parameters.BaseUrl.

Parameters (the openai type). The OpenAI REST-spec sampling fields are typed and all optional - null omits the field, so the upstream's own default applies; everything outside the spec rides Extra.

KeyDefaultNotes
BaseUrlhttp://localhost:8000Server base URL, without /v1.
ModeldefaultThe upstream model id, sent as model on /v1/chat/completions. Must exist on the server (GET <BaseUrl>/v1/models).
ApiKey(unset)Secret - env / user-secrets only, never appsettings.json. Sent as Authorization: Bearer. Empty for a keyless vLLM.
RequestTimeoutSeconds120Per-item pre-stream timeout for this provider (time to response headers) - not the stream duration, which the turn budget bounds. Each provider gets its own named HTTP client.
RetryCount2Per-item pre-stream retries for this provider. Safe for chat - a retried attempt means no tokens were streamed. 0 disables.
Temperature ...(unset)Typed OpenAI-spec sampling (Temperature, TopP, PresencePenalty, FrequencyPenalty, Seed, Stop). Each unset field is omitted so the upstream default stands.
Extra{}A string->string map for everything outside the OpenAI REST spec, keyed by dotted JSON path. Home of the vLLM extensions (top_k, min_p, repetition_penalty) and the template kwargs (chat_template_kwargs.enable_thinking, chat_template_kwargs.preserve_thinking).

Thinking vs instruct is a provider choice. There is no per-request thinking toggle - you pick a thinking provider or an instruct provider. The same physical model can appear under several slugs with different sampling:

"Chat": {
  "DefaultProvider": "qwen36-thinking",
  "Providers": {
    "qwen36-thinking": {
      "Name": "Qwen 3.6 - thinking",
      "Type": "openai",
      "Capabilities": [ "tools", "vision" ],
      "Context": 131072,
      "Parameters": {
        "BaseUrl": "http://vllm-host:8000",
        "Model": "qwen36",
        "Temperature": 0.6,
        "TopP": 0.95,
        "Extra": { "top_k": "20", "chat_template_kwargs.enable_thinking": "true" }
      }
    }
  }
}

Gert binds this statically at startup and never probes the upstream, so filling in Context and Capabilities is on you. Users pick from this catalog; they can never add endpoints or model ids of their own.

Auth + Storage + Gert:Database + Gert:Rag - identity, the data root, and the engines

KeyNotes
Auth:AuthorityThe OIDC issuer (Pocket ID). JWTs are validated against its JWKS, RS256 only.
Auth:AudienceExpected aud claim, e.g. gert-api.
Storage:DataRootFilesystem root holding the users/ tree - the shared default for the object store and both SQLite engines. Back up this directory (and any override roots) and you've backed up Gert.
Storage:ExpectedIssuerFail-closed iss assertion checked before any user folder is created. Normally equal to Auth:Authority.
Gert:Database:TypeEngine for the per-user/per-project databases (user.db/chat.db). Only Sqlite ships (the default - per-user SQLite files); case-insensitive.
Gert:Database:Parameters:DataRoot(SQLite only, optional) Own filesystem root for user.db + chat.db. Unset -> falls back to Storage:DataRoot.
Gert:Rag:TypeEngine for the per-project RAG/vector index - a separate capability from Gert:Database. Only Sqlite ships (the default - per-project rag.db with sqlite-vec + FTS5).
Gert:Rag:Parameters:DataRoot(SQLite only, optional) Own filesystem root for rag.db. Unset -> falls back to Storage:DataRoot. Set it to put the vector index on its own (larger) volume.
Gert:Rag:Parameters:VecExtensionPath(SQLite only, optional) Path to the native sqlite-vec extension. Unset -> the vendored copy beside the running assembly.

Both Storage values are required - the host refuses to start without them. Gert:Database:Type and Gert:Rag:Type are optional and default to Sqlite. Database, RAG, and storage are independent stores: by default all three sit under Storage:DataRoot, but a SQLite engine can take its own root.

The web-search tool backend. Type selects the implementation (SearXNG only today). The fetch step (downloading result pages) is the SSRF-exposed part and is off by default. The web_fetch tool shares this section's fetch caps (same guarded fetcher).

Gert:Tools:Sandbox - the run_python sandbox

The run_python backend. Type picks one of two implementations behind the one sandbox port: Monty (the default - a Rust no-syscall Python interpreter reached via a sidecar, no container infra) or GVisor (opt-in - an ephemeral runsc container). The cross-backend per-run caps sit beside Type; the per-backend bag lives under Parameters. The defaults are the security posture: egress off, no /data mount, hard caps.

KeyDefaultNotes
TypeMontyMonty (Rust interpreter via the sidecar - no container) or GVisor (runsc container). Case-insensitive; unknown fails fast at startup.
WallClockSeconds10Kill timeout per run (both backends).
MemoryMiB256Memory limit (both backends).
MaxOutputBytes65536Captured stdout/stderr cap (both backends).

Per-tool budgets - Gert:Tools:<toolId>:Limits. Every tool ships concrete intrinsic bounds, so a tool always has a budget; Limits is a partial operator override keyed by the tool id (case-insensitive), so web_search's budget lives under Gert:Tools:Search:Limits and run_python's under Gert:Tools:Sandbox:Limits. The fields are MaxCallsPerTurn (default 64; web_search ships 5), CallTimeout (default 00:01:00), and TokenBudget (default 16384).

Gert:Turn - the detached turn pipeline

A round is one upstream completion request that comes back with tool calls - executing them and re-prompting starts the next round, so every round costs a full vLLM completion. The guards are layered: bound every part, brake the loop, make every trip visible on its tool card. The per-tool budgets no longer live here - each tool ships concrete defaults that an operator retunes under Gert:Tools:<toolId>:Limits.

KeyDefaultNotes
MaxTurnDuration00:05:00Hard wall-clock cap on one turn (model rounds + tools) - the real budget. Doubles as the orphan horizon: a streaming row older than this reads as error.
MaxConcurrentTurns4Global cap on turns running concurrently across all users, projects, and conversations. 1 = global serial worker. Per-conversation serialization is enforced separately. Must be >= 1 (validated at startup).
MaxToolRounds64Runaway brake, not a work budget - sized an order of magnitude above legitimate turns. Past it the runner refuses further calls with budget-exhausted errors (visible on the cards) and winds the turn down.
MaxTokensPerRound16384Per-round completion bound: the max_tokens sent on every upstream request. Reasoning tokens count against it on a thinking provider - keep it generous. 0 disables.
AskUserTimeout00:05:00How long one ask_user question waits for the user before the tool returns its graceful "user did not respond" result. The effective wait is min(this, remaining turn budget - 15 s grace), so it can never outlive MaxTurnDuration.
MaxInlineAttachmentContextFraction0.5Largest share of the selected provider's Context (tokens) an inline text-file attachment (dropped into the composer) may consume. A drop estimated over this is refused at send with a 400 steering the user to the Knowledge panel (RAG). Only enforced when the provider declares a Context.
DeltaFlushInterval150msDelta coalescing window - buffered chunks emit as one event per window. 0 disables coalescing.
DeltaFlushMaxChars512Size backstop for the coalescing window.

MaxTokensPerRound is the single per-round max_tokens for every turn - sampling is no longer a user/conversation cascade (it rides the selected provider's Parameters), so there is no per-model cogwheel to override it.

Gert:Prompts - operator-configurable system prompt

System-prompt fragments the host prepends to every turn - not a secret, so they live in appsettings.json. The service layer ships empty defaults; the host binds Gert:Prompts over them.

KeyNotes
CanvasThe canvas/artifact convention prepended first to every turn's system prompt (before a project's pinned instructions). It steers the model to call the make_artifact/edit_artifact/read_artifact tools instead of pasting whole files into code blocks. Empty omits it. Editing it invalidates the vLLM prefix cache for all conversations at once - it is the first bytes of every rendered prompt.

Artifacts - served-artifact tickets

The separate-origin HTML-artifact preview and the HMAC-signed capability URLs it rides (security F3). A top-level section, like Auth/Storage.

KeyDefaultNotes
Origin(empty)The separate origin (scheme://host[:port]) that serves rendered HTML artifacts - a sandbox subdomain in prod, a second port in dev/CI. Empty means same origin: isolation rests on the iframe sandbox alone.
Secret(unset)Secret - env / user-secrets only. HMAC signing key for ticket URLs. An explicit value must be at least 32 UTF-8 bytes. Unset = a random per-process key: fine for one instance, but tickets stop surviving restarts and instances behind a LB won't accept each other's tickets.
Lifetime00:05:00Ticket validity window. Long enough to load the iframe, short enough that a leaked URL is near-useless.

Gert:RateLimiting - the per-user API limiter

A fixed window per user (security F10): each authenticated caller gets its own partition keyed by the token (iss, sub) pair - the same identity anchor as the user folder key (anonymous traffic falls back to the remote IP). One client - or one stolen token - can't saturate the box, and one user's bursts never throttle another's. Applied to /api/* only; /healthz is exempt. A rejected request is a branded 429. Leave the section absent and nothing changes.

KeyDefaultNotes
PermitLimit600Max requests per partition (per user / per anonymous IP) within one window.
Window00:01:00The fixed window length.

Logging - verbosity

The standard .NET Logging:LogLevel config drives Serilog (the host logger): set the floor with Logging:LogLevel:Default (Trace|Debug|Information|Warning|Error|Critical), and quiet individual categories with overrides such as Microsoft.AspNetCore: Warning. Booting at Debug makes the turn pipeline and the HTTP traffic to the model upstream visible; the OpenAIWireLogger then traces the exact POST sent upstream (the api-key bearer redacted). The wire trace carries message content, so it is a local tuning mode only - never run the host at Debug in production.
