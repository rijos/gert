Security Model

One-line posture: isolation is structural - token -> folder, no cross-user query exists - so the residual attack surface is content the box renders or fetches. Most controls harden that surface.

Trust boundaries

  1. Browser <-> API - everything from the browser is untrusted, including a valid token's request body. Bearer-in-header (not cookies) means no CSRF; single origin means no CORS surface.
  2. User <-> user / project <-> project - enforced by the filesystem, not a query filter. The one boundary that can't be "forgotten in a WHERE clause."
  3. API <-> external content - web pages, search results, and uploaded files are untrusted content the server parses or fetches. The SSRF / XXE / parser surface.
  4. Model output <-> browser - LLM text, Markdown, and HTML/SVG artifacts are untrusted output the browser renders. The XSS surface.

Authentication

The SPA is a public OIDC client using Authorization Code + PKCE against Pocket ID, which authenticates users with passkeys (WebAuthn) - there are no passwords, and the API never sees credentials. It only ever fetches the IdP's JWKS to validate signatures.

ClaimUsed for
subThe user key -> folder name. Stable, opaque, never recycled.
groupsAuthorization - the gert-admins group gates the admin policy.
gert_toolsTool entitlement - which tools this user may ever invoke ("rag search sandbox" or "*").
iss / aud / expValidation - issuer, audience, and lifetime.

Residual risks, accepted knowingly

  • Prompt injection is self-scoped. A malicious document can steer this user's turn, but can only touch the user's own data and own-entitlement tools. The exfiltration channels (web fetch, sandbox egress) are off by default.
  • ~1-hour revocation window. There is no token denylist (it would break multi-instance); off-boarding is effective within the IdP's access-token lifetime.
  • Trusted-operator admin. The admin surface is enumerate + delete, gated by the group claim and the {key} validation above.
