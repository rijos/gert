Chat, RAG & Tools

Gert speaks the OpenAI-compatible chat protocol with function calling and streaming, so any vLLM-served model with tool support drives the full loop.

The thirteen tools

Gert advertises up to thirteen built-in tools to the model. The core set:

ToolWhat it does
search_documentsHybrid search over this project's private documents and memory.
read_documentReturn a stored document's full text (paged by offset/max_chars) when ranked passages aren't enough.
web_searchSearch the web via SearXNG; optionally fetch + summarize results (SSRF-guarded).
web_fetchFetch one model-named public URL and return its readable text (same SSRF guard).
run_pythonExecute Python in a sandbox; only stdout/stderr returns (see The sandbox).
set_todosReplace the model-managed todo checklist the chat window renders live.
get_datetimeCurrent date/time, user-local by default (UTC + optional IANA timezone).
run_sub_agentDelegate one self-contained task to a nested agent and wait for its final answer only.
ask_userAsk up to four clarifying questions mid-turn (rendered as tabs) and block until answered.

The remaining four are the canvas suite - make_artifact, edit_artifact, read_artifact, and list_artifacts - that round the roster out to thirteen (see Artifacts).

Tool specs are a token budget. The tools array renders into the prompt, and a model's tool-call format adherence collapses once that region grows too large, so every description stays one or two short sentences carrying only the behavioural contract. Each tool's schema is generated from its typed argument record, not hand-written, so the advertised shape can't drift from the parsed contract.

Only tools that are (a) granted by the user's gert_tools JWT entitlement, (b) enabled on the conversation, and (c) requested in the message are offered to the model. The entitlement is the hard ceiling - enforced when tools are advertised and re-checked at execution time.

Hybrid retrieval

search_documents runs two queries against the project's rag.db and fuses them:

  1. Vector KNN against vec_chunks - the query is embedded by the same model that embedded the chunks.
  2. Lexical BM25 against fts_chunks.
  3. Reciprocal Rank Fusion merges the two ranked lists into the final top-k, joined back to chunks and documents for content, page, and filename.

The result feeds the tool card and seeds the inline citations ([1] -> file.pdf - p.4).

Memory rides the same query. A project's memory entries are embedded into the same rag.db, distinguished only by kind = 'memory' - so retrieval covers documents and memory together with no extra plumbing. Pinned memory is additionally prepended to the system prompt.

Artifacts (canvas tabs)

Artifacts are created by explicit tool calls, not by parsing the model's prose. (An earlier convention - a named fenced code block - was replaced wholesale: as opaque tool arguments the content can't be truncated by a file's own fences.) Four functions drive the canvas:

ToolWhat it does
make_artifactCreate (or overwrite by name) a complete file. format is a closed set: html, markdown, svg, python, csharp, cpp, javascript, rust.
edit_artifactIterate by exact substring replacement (old_str must match exactly and once), mirroring the str_replace contract.
read_artifactReturn an artifact's current content, line-numbered, so a follow-up edit can copy a snippet verbatim.
list_artifactsEnumerate the conversation's artifacts (name, format, version) so the model picks the right one instead of guessing.

Each call persists as a tool_calls row; created/updated artifacts open as canvas tabs live, mid-turn - with HTML preview, source view, and download. A reload gets the same artifacts back through the thread GET. The operator-configurable Gert:Prompts:Canvas fragment rides each turn's system prompt and tells the model to call make_artifact for any complete file instead of pasting a fenced block; inline code blocks stay inline in the message.

Detached turns

A turn is detached from the request that started it. POST .../messages only plans - validate, persist the user message and a streaming placeholder, snapshot identity and entitlements - then enqueues and responds 202 Accepted. A background worker runs the tool loop:

  • Every event gets a per-conversation monotonic seq and is persisted before it is published - the database is the source of truth, transports are just delivery.
  • Clients attach over SSE or range polling and switch freely - the seq watermark makes it gap- and duplicate-free.
  • Generation survives disconnects: a reload mid-turn resubscribes from the streaming row's seq and loses nothing.
  • POST .../cancel stops the in-flight turn server-side; the partial answer persists as cancelled.

Document ingestion

Upload returns 202 immediately; a background worker does the rest:

1. extract   # PDF -> PdfPig, DOCX -> OpenXML - in an isolated subprocess
2. chunk     # token-aware windows with overlap; page/section tracked
3. embed     # batched against the vLLM embeddings endpoint
4. write     # chunks + vec_chunks + fts_chunks; progress emitted
5. status    # processing -> ready | failed

Hardened extraction. Uploads are untrusted bytes fed to large parsers, so extraction runs out-of-process in an unprivileged, resource-capped helper: dropped privileges, no network, read-only access to the one input file, RLIMIT_AS/CPU/NPROC + wall-clock timeout, DTD/external entities off (XXE), and decompressed-size caps (zip bombs). A crash fails that document, never the host.

The sandbox

run_python runs untrusted, model-written code - the strongest blast radius in the system. Two backends sit behind one port, chosen by the operator (Gert:Tools:Sandbox:Type):

  • Monty (default) - a minimal Python interpreter written in Rust with no syscalls: the language itself has no filesystem, network, or env access, so untrusted code can only reach the world through host callbacks (plain run_python grants none). It runs in an unprivileged sidecar process Gert reaches over HTTP - microsecond startup, no container, no infra. It supports a Python subset; an unsupported construct returns an error the model reads.
  • gVisor (runsc) (opt-in) - an ephemeral container running real CPython, for workloads needing the full language and stdlib: no inbound network, outbound off by default, read-only rootfs, a small writable /tmp, container destroyed after the call. Needs gVisor on the host.

Both backends share the per-run posture: no mount of /data (the sandbox never sees any user's - or this user's - database files), a hard wall-clock and memory cap, and only captured stdout/stderr returns.
