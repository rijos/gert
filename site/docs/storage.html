Storage & Isolation

Data is owned by the filesystem at two levels: each user is a folder, and within it each project is a folder with its own databases - fully isolated for the same structural reason.

Layout

/data/
└── users/
    └── {key}/                     # key = sha256(iss + sub)
        ├── meta.json              # identity - iss, sub, username, created_at
        ├── settings.json          # theme, language, defaults
        └── projects/
            ├── default/           # lazily created; always present
            │   ├── meta.json      # name, instructions, defaults
            │   ├── chat.db        # conversations, messages, tool calls, artifacts
            │   ├── rag.db         # sqlite-vec: documents, chunks, embeddings, FTS
            │   ├── files/         # original uploaded files
            │   └── memory/        # memory entries (markdown)
            └── {project-id}/      # same shape, fully isolated

Why the key derivation matters

The folder key is computed as sha256(iss + "\n" + sub) from the validated JWT on every request - never from a route parameter, query string, or body. Consequences:

  • IDOR is structurally impossible - a query physically cannot reach another user's data, because the connection only ever opens this user's files.
  • Identifier reuse is closed - the anchor is sub (the IdP's stable, never-recycled UUID), not email or username, so a reassigned identifier can't inherit someone else's folder.
  • Traversal-proof - the hash yields a fixed-length, path-safe folder name for any claim value the IdP emits.

Project ids do come from the request path, but a pid is validated to a UUID or the literal default and only ever joined under the token-derived user folder - it selects among this user's projects and can never escape.

Splitting keeps a vector-index rebuild from locking chat history, and lets a user clear a project's knowledge base without losing its conversations.

Connections & concurrency

PRAGMA journal_mode = WAL;
PRAGMA busy_timeout = 5000;
PRAGMA foreign_keys = ON;
  • WAL lets the ingestion worker write rag.db while the user's query reads it.
  • Connections are open-per-request (open -> use -> dispose); Microsoft.Data.Sqlite pools handles internally.

Lazy provisioning & migrations

On the first authenticated request, provisioning runs fail-closed and before any disk access:

  1. Validate the identity before touching disk - assert iss equals the configured authority, aud equals the audience, and sub is well-formed. No valid identity -> no folder.
  2. Create the user root, settings.json, and the default project.
  3. Per project, open each database, check PRAGMA user_version, and apply any newer plain-SQL migration scripts.

There is no global migration step because there is no global database.

Lifecycle

  • Delete a project -> rm -rf projects/{pid} (the default project is emptied, not removed).
  • Forget documents -> wipe one project's rag.db + files/, keep its chats.
  • Delete a user -> rm -rf users/{key}, plus revocation in the IdP.
  • Back up Gert -> back up one directory.
