Architecture

Five moving parts, one origin, zero shared state.

ComponentRole
VanJS SPAThe UI, served as static files by Gert.Api itself. Holds the access token in memory, calls /api/* on the same origin - no CORS needed.
Pocket IDOIDC provider. Authenticates users with passkeys, issues JWTs, owns the user list. The API only ever fetches its JWKS.
Gert.ApiStateless REST API + static host. Validates JWTs, derives the user folder, orchestrates chat and ingestion.
vLLMChat + embedding models behind an OpenAI-compatible API. Server-side only.
SearXNGSelf-hosted web search. Server-side only.
gVisorEphemeral sandbox for the run_python tool. Server-side only.

Design principles

  1. No central application database. No users table. Identity lives in the IdP; per-user data lives on disk. The API is a stateless translator between JWTs and folders.
  2. Isolation is a filesystem property, not a query filter. A connection only ever opens this user's files - there is no WHERE user_id = ? to forget. The same property nests one level down for projects.
  3. The user key is derived only from the token. sha256(iss + sub) from the validated JWT, never from a route, query, or body - IDOR is structurally impossible.
  4. Lazy provisioning. Folders and schemas are created on first authenticated request. A valid token is sufficient to exist.
  5. Deletion is rm -rf. Nothing references a user outside their folder.
  6. Validation is a fail-closed input boundary. Every user-supplied DTO passes a registered validator before any service acts on it; an input type with no validator is a bug, not an exemption.

Solution layout

The compiler enforces inward dependencies - Gert.Service references only Gert.Model, so the host (API) and the tests drive the same business logic:

src/
├── Gert.Model/             # POCOs only - no dependencies
├── Gert.Service/           # host-agnostic business logic
│   ├── Chat/               # turn planner + runner (the tool loop)
│   ├── Ingestion/          # extract -> chunk -> embed -> write
│   ├── Tools/              # tool registry (RAG, search, sandbox...)
│   └── Validation/         # fail-closed input boundary
├── Gert.Storage/           # object store (local FS today)
├── Gert.Database/          # DB provider seams
├── Gert.Database.Sqlite/   # sqlite-vec + FTS5 implementation
├── Gert.Authentication/    # JWT claims -> IUserContext
├── Gert.External/          # vLLM, SearXNG, gVisor, extraction adapters
└── Gert.Api/               # REST host + the SPA (wwwroot/)

Tech stack

  • .NET 10 / ASP.NET Core - warnings as errors, nullable strict.
  • SQLite via Microsoft.Data.Sqlite + Dapper, with sqlite-vec (vector KNN) and FTS5 (BM25) per project database.
  • VanJS SPA - native ES modules, vendored libs, no npm, no build step; minified in place on publish.
  • Polly resilience on all upstream HTTP; Serilog NDJSON logs (no tokens, no content).
  • Channels + BackgroundService for the detached turn and ingestion workers.
