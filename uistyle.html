firecrawl.dev — Best Vector Databases 2026

Short version: sqlite-vec is enough for your setup, and it happens to make per-user isolation almost free.

Because you're giving every user their own rag.db file, each corpus stays small, so the brute-force vec0 scan is plenty fast and a query physically cannot cross users1. Qdrant only starts to earn its keep once a single shared index grows into the millions of vectors — not your situation2.

Keep Qdrant on the shelf as a one-line swap if a corpus ever outgrows the file

Vector store: the call

Recommendation: ship with sqlite-vec. Revisit only if a single corpus crosses ~1M vectors.

Why

  • One rag.db per user → isolation is a filesystem property, not a query filter you can forget to apply.
  • Corpora stay small, so brute-force vec0 scans stay well under 10 ms.
  • Zero extra services to run, back up, or monitor on the box.

# Vector store: the call

**Recommendation:** ship with `sqlite-vec`. Revisit only if a
single corpus crosses ~1M vectors.

## Why
- One `rag.db` per user → isolation is a *filesystem* property.
- Corpora stay small, so `vec0` scans stay under 10 ms.
- Zero extra services to run, back up, or monitor.

## When Qdrant wins
| Scenario                  | sqlite-vec | Qdrant     |
|---------------------------|------------|------------|
| < 100k vecs / user        | ideal      | overkill   |
| shared multi-tenant index | —          | built for  |
| HNSW + payload filters    | —          | native     |

> Keep the `VectorStore` interface thin so the swap is one line.
