# Core design principles

1. **No central application database.** There is no `users` table in the API. Identity lives in Pocket ID; per-user data lives on disk. The API is a stateless translator between JWTs and folders.
2. **Isolation is a filesystem property, not a query filter.** Each user gets a private directory containing their own SQLite files. A query physically cannot reach another user's data because the connection only ever opens *this* user's files. There is no `WHERE user_id = ?` you can forget to apply — this is the same point the mockup makes about per-user `rag.db`. The same property **nests one level down**: each project is its own folder, so projects are isolated from one another for exactly the same reason ([configuration → projects](configuration.md#2-projects)).
3. **The user key is derived only from the token.** The folder is resolved as `sha256(iss + sub)` from the validated JWT on every request — *never* from a route parameter, query string, or body. This makes IDOR (one user reading another's data) structurally impossible. The anchor is **`sub`** (the IdP's stable, never-recycled UUID), not email or username, so an identifier can't be *reused* onto someone else's folder; the identity is validated **before any disk access**, and past that gate the validated JWT is trusted ([decisions §3](decisions.md#3-folder-key), [security F12](security.md#3-findings--remediations)).
4. **Lazy provisioning.** A user's folder and database schema are created on first authenticated request, not by an admin step. A valid token is sufficient to exist.
5. **Deletion is `rm -rf`.** Because nothing references a user outside their folder, removing a user is removing their directory (plus revoking them in the IdP — see [Operations → User lifecycle](operations.md#user-lifecycle--remove-a-user--remove-a-folder)). The same holds at project scope: deleting a project is deleting its folder, and "forget my documents" is clearing one project's `rag.db` ([configuration → data lifecycle](configuration.md#5-data-lifecycle-user-facing)).
6. **Validation is a fail-closed input boundary.** Every user-supplied DTO passes through `IValidationProvider` in the service layer *before* a service acts on it — so the API and the Console are guarded by the same rules, with no unvalidated back door. The default is closed: an input type with no registered `IValidator<T>` is a bug, not an exemption (a [fail-closed test](testing.md#validation--the-input-security-boundary) keeps it that way). This is defence-in-depth, not the perimeter — isolation (#2, #3) and parameterized SQL still stand on their own; validation rejects malformed or abusive *content* before it reaches a service or the disk.
