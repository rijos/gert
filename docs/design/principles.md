# Core design principles

1. **No central application database.** There is no `users` table in the API. Identity lives in Pocket ID; each user's data lives in its own per-user stores. The API is a stateless translator between JWTs and those stores.
2. **User data is isolated by the database, not by an application query filter.** Each user's data - structured and vector alike - is reached only through a store opened from the token-derived key (`sha256(iss + sub)`), so a query cannot stray to another user and there is no `WHERE user_id = ?` to forget. How an engine partitions per user varies - a database/file, a schema, a collection, with any cross-user shared data in its own shared partition - but **scoping always lives in the store, keyed from the token, never in an application query.** It **nests one level down**: each project opens as its own scope ([configuration -> projects](configuration.md#2-projects)).
3. **The user key is derived only from the token.** No storage backend - a filesystem, an object store (S3/Azure Blob), a database, a vector index - has a native notion of "this user," so identity is reduced to one **stable, store-agnostic key**: `sha256(iss + sub)`, computed from the validated JWT on every request, *never* from a route parameter, query string, or body. That key is what every store scopes by - a directory name, an object-key prefix, a namespace, a partition - so IDOR (one user reading another's data) is structurally impossible. The anchor is **`sub`** (the IdP's stable, never-recycled UUID), not email or username, so an identifier can't be *reused* onto someone else's data; the key is validated **before any store is touched**, and past that gate the validated JWT is trusted ([decisions section 3](decisions.md#3-user-key), [security F12](security.md#3-findings--remediations)).
4. **Lazy provisioning.** A user's folder and database schema are created on first authenticated request, not by an admin step. A valid token is sufficient to exist.
5. **Deletion is each store erasing a scope, not a row-scrub the application maintains.** Nothing references a user outside their own stores, so erasing a user is each store dropping that user's partition - the **service** sequences it (the databases, then the object store) and revokes them in the IdP ([Operations -> User lifecycle](operations.md#user-lifecycle---remove-a-user)). There is no central cross-entity table to scrub and no application-level filtered `DELETE` to forget - the engine scopes the erase just as it scopes a read (#2), the contrast that rules out shared-tables+RLS ([tech-stack](tech-stack.md)). The same holds per project: deleting a project drops its stores together, and "forget my documents" clears one project's RAG index ([configuration -> data lifecycle](configuration.md#5-data-lifecycle-user-facing)). Spanning independent stores, the erase is made **crash-consistent** by a write-ahead deletion journal + idempotent replay - an interrupted delete resumes to completion, never leaving orphans ([decisions section 12](decisions.md#12-deletion-crash-consistency---a-journal--idempotent-forward-recovery)).
6. **Validation is a fail-closed input boundary.** Every user-supplied DTO passes through `IValidationProvider` in the service layer *before* a service acts on it - so every caller is guarded by the same rules, with no unvalidated back door. The default is closed: an input type with no registered `IValidator<T>` is a bug, not an exemption (a [fail-closed test](testing.md#validation---the-input-security-boundary) keeps it that way). This is defence-in-depth, not the perimeter - isolation (#2, #3) and parameterized SQL still stand on their own; validation rejects malformed or abusive *content* before it reaches a service or the disk.
