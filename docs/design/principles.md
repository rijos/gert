# Core design principles

1. **No central application database.** There is no `users` table in the API. Identity lives in Pocket ID; per-user data lives on disk. The API is a stateless translator between JWTs and folders.
2. **Isolation is a filesystem property, not a query filter.** Each user gets a private directory containing their own SQLite files. A query physically cannot reach another user's data because the connection only ever opens *this* user's files. There is no `WHERE user_id = ?` you can forget to apply — this is the same point the mockup makes about per-user `rag.db`.
3. **The user key is derived only from the token.** The folder is resolved from the JWT `sub` claim on every request. It is *never* taken from a route parameter, query string, or body. This makes IDOR (one user reading another's data) structurally impossible.
4. **Lazy provisioning.** A user's folder and database schema are created on first authenticated request, not by an admin step. A valid token is sufficient to exist.
5. **Deletion is `rm -rf`.** Because nothing references a user outside their folder, removing a user is removing their directory (plus revoking them in the IdP — see [Operations → User lifecycle](operations.md#user-lifecycle--remove-a-user--remove-a-folder)).
6. **Validation is a fail-closed input boundary.** Every user-supplied DTO passes through `IValidationProvider` in the service layer *before* a service acts on it — so the API and the Console are guarded by the same rules, with no unvalidated back door. The default is closed: an input type with no registered `IValidator<T>` is a bug, not an exemption (a [fail-closed test](testing.md#validation--the-input-security-boundary) keeps it that way). This is defence-in-depth, not the perimeter — isolation (#2, #3) and parameterized SQL still stand on their own; validation rejects malformed or abusive *content* before it reaches a service or the disk.
